Goal

Fix PlantUML sequence diagram column ordering so parent columns (and nested group/participant ordering within those parents) follow first appearance in the dynamic view interaction sequence (i.e., the order relationships introduce participants), rather than sorting by element id/name.

Why Only One Test Changed (Expectation)

- Only tests that assert full PlantUML text for a sequence diagram AND include multiple parent boundaries (software systems/containers/groups) with an order that differs between "sorted" vs "first appearance" will change.
- Many other tests either:
  - don’t render as sequence diagrams,
  - don’t include multiple parents,
  - or don’t assert ordering-sensitive text (or happen to match the old sort order).

Regenerating JSON Fixtures (What/Why)

The export tests commonly load workspaces from JSON fixtures under `structurizr-export/src/test/resources/*.json`.
Those fixtures are derived from DSL examples (where available) and should be kept in sync when:

- The DSL source changes.
- The DSL parser/serialization changes.
- We want deterministic relationship/element ordering as stored in JSON (to match “first appearance” semantics across different sources).

Known fixture pairs in this repo:

- `structurizr-export/src/test/resources/sequence.dsl` -> `structurizr-export/src/test/resources/sequence.json`
- `structurizr-export/src/test/resources/groups.dsl` -> `structurizr-export/src/test/resources/groups.json`
- `structurizr-export/src/test/resources/amazon-web-services.dsl` -> `structurizr-export/src/test/resources/amazon-web-services.json`
- `structurizr-dsl/src/test/resources/dsl/big-bank-plc.dsl` -> `structurizr-export/src/test/resources/big-bank-plc.json`

Implementation Plan

1) Confirm scope of ordering change

- Identify the PlantUML sequence diagram generation path and the exact data structure used for ordering (dynamic view relationships in declared order).
- Ensure the ordering rules apply to:
  - software system boundaries (dynamic view w/ container scope)
  - container boundaries (dynamic view w/ component scope)
  - groups (nested groups via `structurizr.groupSeparator`)
  - participants within a group

2) Implement “first appearance” ordering for DynamicView

- Add a helper that scans `DynamicView.getRelationships()` in order and assigns an incrementing index to each element id at first sight (source then destination).
- Use that index to order:
  - boundary software systems/containers by the earliest index of any child element rendered under that boundary.
  - group ordering by the earliest element occurrence that belongs to the group; preserve nested group parent-before-child.
  - participant ordering within a group by the first appearance index.
- Keep existing stable behavior for non-DynamicView exports (continue sorting by id/name as before).

Primary file likely touched:

- `structurizr-export/src/main/java/com/structurizr/export/AbstractDiagramExporter.java`

3) Update ordering-sensitive tests

- Update the expected PlantUML strings in tests that assert the full sequence diagram definition.
- Ensure tests cover at least one case where first appearance differs from id/name sort.

Primary file likely touched:

- `structurizr-export/src/test/java/com/structurizr/export/plantuml/StructurizrPlantUMLDiagramExporterTests.java`

4) Add a repeatable JSON fixture regeneration mechanism

Because there is no existing Gradle task/script in the repo for this, add one of:

- Option A (Recommended): a small Java utility under `structurizr-export/src/test/java/...` (or `src/testFixtures/java` if present) that:
  - parses DSL using `com.structurizr.dsl.StructurizrDslParser`
  - writes JSON using `com.structurizr.util.WorkspaceUtils.saveWorkspaceToJson`
  - takes input/output paths
  - optionally pretty-prints to keep diffs readable

- Add a Gradle task in `structurizr-export/build.gradle` (JavaExec) to run it, e.g.:
  - `./gradlew :structurizr-export:regenerateTestWorkspaces`

Chosen: add the task and commit regenerated JSON fixtures in this PR.

The task should regenerate, at minimum:

- `sequence.json` from `sequence.dsl`
- `groups.json` from `groups.dsl`
- `amazon-web-services.json` from `amazon-web-services.dsl`
- `big-bank-plc.json` from `structurizr-dsl/src/test/resources/dsl/big-bank-plc.dsl`

After regeneration, commit updated JSON outputs under `structurizr-export/src/test/resources/`.

5) Verification

- Regenerate fixtures: `./gradlew :structurizr-export:regenerateTestWorkspaces`
- Run module tests: `./gradlew :structurizr-export:test`
- Run full suite: `./gradlew test`

Notes / Edge Cases

- If a boundary/group has no elements that appear in the relationship sequence (e.g., added explicitly to the view but never participates), define a deterministic fallback (e.g., keep id sort after “infinite” index).
- Relationship order must be preserved as declared in DSL/JSON; ensure the regeneration step does not reorder relationships.
